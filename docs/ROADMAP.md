# Polymarket Bot - Роадмап и Архитектура

## Основная стратегия
Бот на Python для Polymarket. Стратегия: систематически искать и выкупать дешёвые доли на маловероятные исходы ("чёрные лебеди") в нишах с низкой эффективностью ценообразования. Это не попытка "угадывать разворот" и не классический buy-low-sell-high по индикаторам.

Мы эксплуатируем структурную особенность бинарных рынков: когда исход кажется почти невозможным, цена упирается в пол. В этот момент downside уже почти исчерпан, а upside может быть кратным. Большинство входов сгорит, но редкие сильные отскоки окупают серию мелких потерь.

### Что именно мы ищем
- токены, которые торговались в зоне экстремально низких цен;
- при этом имели реальную ликвидность на дне, а не только "бумажный" принт;
- и исторически показывали последующий значимый отскок;
- желательно в категориях с высокой информационной асимметрией.

### Подтверждение стратегии (реальный кейс)
Трейдер **planktonXD** (Polymarket, февраль 2025 — февраль 2026): из ~$1,000 стартового капитала через 72,533 автоматизированных предсказаний получил **$98,000+ прибыли**. Крупнейший одиночный выигрыш — $2,527 (лишь 2% от общей прибыли). Профит-кривая — стабильные 45° вверх без значимых просадок. Примеры: 3,664 акции победы Fuego по **0.1 цента** → +$874 (+23,750%); $16 на падение SOL до $130 по **0.7 цента** → +$1,574 (+9,285%).

### Ключевые ниши (высокая информационная асимметрия)
- **Киберспорт** — региональные лиги, мелкие турниры, раунды внутри матчей
- **Региональная политика** — выборы в малых странах, местные референдумы
- **Крипто-ценовые триггеры** — "достигнет ли X цены Y в период Z"
- **Погода и гиперлокальные события** — малоизвестные рынки с минимальной аудиторией

## Роадмап (План разработки)

### Этап 1. Исторический датасет и бэктестинг
- Изучение API Polymarket: Gamma API, CLOB API и data-api trades. Рабочая выжимка: [API_REFERENCE.md](./API_REFERENCE.md). Ограничения и троттлинг: [API_LIMITS.md](./API_LIMITS.md).
- Сбор закрытых рынков через `GET /markets` (Gamma API) с мягкой фильтрацией мусора на уровне рынков.
- Для каждого рынка скачивание **сырых трейдов**, а не `prices-history`.
- Базовый рабочий фильтр для истории трейдов: `filterType=CASH&filterAmount=10`, чтобы уместиться в лимит пагинации и не забить датасет ботошумом.
- Цель этапа: эмпирически вывести реальные параметры стратегии из данных, а не зафиксировать их заранее.

### Этап 2. Построение аналитической модели лебедей
- Отказ от агрегированных свечей и `price_history`: они сглаживают spikes и скрывают реальные экстремумы.
- Построение таблицы событий `token_swans` на основе **сырых трейдов**.
- Использование Zigzag-подхода:
  - одно событие = локальный минимум → следующий значимый локальный максимум;
  - значимым считается движение минимум в `5x`;
  - один токен может содержать несколько отдельных событий-лебедей.
- Для каждого события считаются:
  - цена входного минимума,
  - объём и число сделок в зоне входа,
  - цена выходного максимума,
  - объём и число сделок в зоне выхода,
  - `possible_x` как отношение `exit_max_price / entry_min_price`.
- Отдельная задача этапа: отделить реальные лебеди от артефактов пустого стакана и микроцен.

### Этап 3. Исследование EV и фильтрация рабочих сетапов
- Сравнение дней, категорий и диапазонов ликвидности по частоте и качеству лебедей.
- Анализ распределений `5x / 10x / 20x / 30x / 50x`.
- Поиск ответа на ключевые вопросы:
  - сколько реальных лебедей происходит в день;
  - где проходит граница между торгуемым дном и пустым стаканом;
  - какие категории и диапазоны ликвидности дают лучший EV;
  - сколько времени цена проводит в зоне входа.
- Результат этапа: стартовые пороги для скринера и риска, основанные на фактах, а не гипотезах.

### Этап 4. Скринер рынков (Market Scanner)
- *Детали архитектуры: [TRADING_LOGIC.md](./TRADING_LOGIC.md) и [RUNTIME_FLOW.md](./RUNTIME_FLOW.md)*
- Скринер работает через **REST polling раз в 5-10 минут** (Gamma API / CLOB API).
- Стратегия не требует HFT: мы не обязаны ловить сам тик падения, нам нужны рынки, которые уже находятся в рабочей зоне входа.
- Скоринг рынка строится на признаках, выведенных из бэктеста:
  - категория,
  - ликвидность,
  - время до закрытия,
  - глубина стакана,
  - статистическая полезность подобных сетапов в прошлом.
- WebSocket используется только там, где он действительно даёт ценность — например, для мониторинга собственных ордеров и позиций.

### Этап 5. Исполнение ордеров (Execution)
- *Детали: [TRADING_LOGIC.md](./TRADING_LOGIC.md). Готовые решения: [BEST_PRACTICES.md](./BEST_PRACTICES.md).*
- Покупка через `py-clob-client` лимитными ордерами.
- Перед входом обязательна проверка книги заявок (`GET /book`) и реальной исполнимости цены.
- Сразу после исполнения выставляется лимитный TP SELL.
- Риск-менеджмент строится вокруг микро-позиций и контроля slippage, а не вокруг "уверенных" ставок.

### Этап 6. Мониторинг позиций и Profit Module
- *Детали: [TRADING_LOGIC.md](./TRADING_LOGIC.md), [RUNTIME_FLOW.md](./RUNTIME_FLOW.md)*
- **Position Monitor:** REST polling `GET /orders` раз в 1-2 минуты. Следит за статусом TP и завершением рынка.
- **Profit Module:** обновляет веса категорий и сетапов по реальным результатам, а не только по историческому бэктесту.
- Логирование и алерты в Telegram: [LOGGING_AND_ALERTS.md](./LOGGING_AND_ALERTS.md).

### Этап 7. Dry Run и деплой
- Полный режим **Dry Run** без реального USDC: скринер, отбор, симуляция входов и выходов, Profit Module.
- Затем запуск на небольшом депозите.
- После валидации — деплой на VPS для круглосуточной работы.

## Что уже выяснено на практике
- `prices-history` нельзя использовать для этой стратегии: он сглаживает внутричасовые экстремумы и скрывает реальные spikes.
- Публичный `/trades` имеет жёсткий лимит: фактически доступны только последние `4000` сделок (`limit 1000`, `max offset 3000`), а фильтры по времени работают ненадёжно.
- Поэтому исторический анализ возможен только при разумной фильтрации шума, в первую очередь через `CASH >= 10`.
- Математические `100x+` на микроценах чаще всего оказываются артефактами пустого стакана; реальные торгуемые лебеди живут скорее в диапазоне до `~50x`.
- Один токен может переживать несколько независимых событий-лебедей; плоская модель "один токен = один summary" теряет важную структуру.

## Архитектура (Структура проекта)
*Используемые библиотеки и готовые решения: [BEST_PRACTICES.md](./BEST_PRACTICES.md). API-справочник: [API_REFERENCE.md](./API_REFERENCE.md).*

- `config.py` — настройки стратегии и пороги.
- `data_collector/` — скачивание рынков, трейдов, парсинг и аналитика.
  - `collector.py` — сбор закрытых рынков и связанных файлов.
  - `parser.py` — нормализация и загрузка данных в SQLite.
  - `analyzer_v2.py` — построение `token_swans` из сырых трейдов.
  - `state_db.py` — журнал состояния пайплайна.
- `api/`
  - `gamma_client.py` — работа с публичным API рынков.
  - `clob_client.py` — работа с книгой заявок и биржевой частью.
- `strategy/`
  - `screener.py` — поиск рынков в зоне потенциального входа.
  - `niche_ranker.py` — ранжирование категорий и ниш.
  - `risk_manager.py` — сайзинг позиций и логика TP.
- `execution/`
  - `order_manager.py` — формирование и отправка ордеров.
- `utils/`
  - `logger.py` — логи.
  - `notifier.py` — алерты.
- `main.py` — точка входа боевого контура.
