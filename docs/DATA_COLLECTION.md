# Сбор исторических данных (Бэктестинг)

Для валидации стратегии покупки дешёвых долей ("чёрных лебедей") перед написанием торгового бота необходимо собрать и проанализировать **сырые исторические данные**. Ключевой принцип: мы не навязываем рынку заранее придуманные константы, а позволяем данным показать, где действительно возникает положительный EV.

## Цель этапа
Собрать датасет закрытых рынков и восстановить из него **реальные события-лебеди**:
- где цена действительно уходила в экстремально низкую зону;
- где на дне была хоть какая-то ликвидность;
- где затем следовал значимый отскок;
- и в каких нишах такие события встречаются достаточно часто, чтобы их можно было торговать системно.

Это исследовательский этап. Константы вроде `entry_price`, `tp_multiplier`, `time_window`, минимальной ликвидности и частоты сканирования должны быть **выведены из данных**, а не выбраны "на глаз".

## Главный архитектурный вывод
**`prices-history` больше не используется.**

Причины:
- он возвращает сглаженные агрегированные точки, а не реальную последовательность сделок;
- он стирает внутричасовые spikes и падения, которые для нас критичны;
- на старых рынках он обрезан и не даёт полной истории;
- в результате он создаёт ложную картину минимумов и максимумов.

**Единственный источник правды для анализа лебедей — сырые `trades`.**

## Пайплайн сбора данных

### 1. Получение списка рынков из Gamma API
Источник: `GET /markets`

Что делаем:
- запрашиваем закрытые рынки (`closed=true`);
- фильтруем по диапазону дат через `end_date_min` и `end_date_max`;
- используем пагинацию `limit` / `offset`;
- сохраняем сырой market JSON без потерь.

### 2. Мягкая фильтрация мусора на уровне рынков
На этом этапе допустима только **операционная фильтрация**, чтобы не тратить часы на откровенный мусор. Это не должна быть жёсткая фильтрация стратегии.

Доступные фильтры:
- `volume_num_min` — минимальный объём торгов в USDC;
- `liquidity_num_min` — минимальная ликвидность;
- `cyom=false` — исключение Create Your Own Market спама;
- `tag_id` — фильтрация по категориям, если нужна отдельная выборка.

Текущие практические замечания:
- `liquidity_num_min` теоретически надёжнее `volume_num_min`, потому что объём легче накрутить;
- `cyom=false` полезен против пользовательского спама, но сам по себе не решает проблему пустых стаканов;
- текущий рабочий порог на стороне коллектора — **`volume_num_min=50`** как мягкий отсев совсем пустых рынков;
- окончательные пороги всё равно определяются не на этом шаге, а SQL-анализом и бэктестом.

### 3. Сохранение сырого market-слоя
Сырые данные по рынкам сохраняются как неизменяемый источник правды:
- в файловую структуру `database/yyyy-mm-dd/{market_id}.json`;
- и/или в SQLite для последующего ETL и пересчётов.

Идея простая: любой следующий аналитический слой должен пересчитываться из сырья, а не из уже агрегированных результатов.

### 4. Скачивание сырых трейдов по каждому рынку
Источник: `data-api.polymarket.com/trades`

Для каждого рынка скачиваются трейды по токенам. Это основной слой данных, на котором строится вся аналитика.

Критически важные детали:
- используем **`filterType=CASH&filterAmount=10`**;
- это не "стратегический" фильтр, а техническая необходимость;
- причина — жёсткое ограничение пагинации: `limit=1000`, `offset <= 3000`, то есть максимум **4000 сделок** на запросную историю;
- без фильтра по cash API забивается микросделками и ботошумом, и мы видим только последние дни перед закрытием рынка;
- фильтр `CASH >= 10` отрезает мелкий шум и позволяет заглянуть существенно глубже в историю.

Дополнительный важный факт:
- публичный `/trades` ненадёжно ведёт себя с фильтрами по времени (`startTs` / `endTs`), поэтому строить на них исторический анализ нельзя.

### 5. Парсинг и нормализация
После скачивания файлов данные разбираются и перекладываются в SQLite.

Назначение этого слоя:
- нормализовать markets / tokens / trades;
- связать токены с рынками и датами;
- подготовить данные для аналитики без повторного чтения тысяч JSON-файлов;
- сделать пайплайн идемпотентным и возобновляемым.

Состояние пайплайна хранится отдельно в `collector_state.db`, чтобы можно было:
- понимать, что уже скачано;
- что уже распарсено;
- где были ошибки;
- безопасно перезапускать обработку.

### 6. Построение событий `token_swans`
Это ключевой аналитический слой проекта.

**`token_swans`** — денормализованная таблица событий-лебедей.
Одна строка = одно событие вида:
- локальный минимум,
- затем значимый рост,
- при наличии реальных сделок в зоне входа и выхода.

Один токен может породить несколько строк, если у него было несколько независимых провалов и отскоков.

## Алгоритм построения `token_swans` (Zigzag)

### Почему старая модель не подошла
Ранние подходы пытались хранить по токену один summary (`min_price`, `max_price`, `spike_multiplier` и т.д.). Это ломает структуру данных:
- токен может несколько раз падать и отскакивать;
- разные события могут иметь разную ликвидность и разную торгуемость;
- один общий summary теряет контекст.

### Новая модель
Используется Zigzag-подход:
- отслеживаются чередующиеся локальные минимумы и максимумы;
- значимым считается только разворот минимум в `5x` (`MIN_RECOVERY=5.0`);
- каждая пара `локальный минимум → следующий локальный максимум` превращается в отдельное событие;
- если в конце истории есть незакрытый, но уже значимый рост, он тоже фиксируется.

### Поля `token_swans`
Для каждого события считаются:
- `entry_min_price` — абсолютный минимум цены в рамках провала;
- `entry_volume_usdc` — суммарный объём сделок в зоне входа;
- `entry_trade_count` — число сделок в зоне входа;
- `entry_ts_first` / `entry_ts_last` — временные границы зоны входа;
- `exit_max_price` — максимум после провала;
- `exit_volume_usdc` — объём сделок в зоне выхода;
- `exit_trade_count` — число сделок в зоне выхода;
- `exit_ts_first` / `exit_ts_last` — временные границы зоны выхода;
- `possible_x` — отношение `exit_max_price / entry_min_price`.

### Определение зон
- **Зона входа:** `price <= entry_min_price * 5`
- **Зона выхода:** `price >= exit_max_price * 0.5`

Это даёт не просто факт "цена когда-то была низкой", а практический профиль события:
- насколько глубоко упал токен;
- была ли ликвидность возле дна;
- был ли реальный объём на отскоке;
- сколько сделок вообще происходило в рабочих диапазонах.

## Ключевые выводы, уже полученные из данных

### 1. Нельзя доверять агрегированным данным для spike-стратегии
Если рынок за один час сходил из `0.001` в `0.18`, а закрыл час по `0.007`, то `prices-history` покажет именно сглаженное закрытие, а не реальный экстремум. Для нашей стратегии это означает потерю самого важного сигнала.

### 2. "100x+" на микроценах чаще всего иллюзия
На практике экстремальные иксы обычно возникают:
- при микроценах ниже реальной исполнимости;
- в пустых стаканах;
- на артефактах последних принтов перед закрытием рынка.

То есть математически `possible_x` может быть огромным, но торгуемо это не всегда. Практический вывод: реальный рабочий потолок обычно ближе к `~50x`, а не к рекламным `500x+`.

### 3. Ликвидность нужно мерить не по рынку в целом, а по зоне события
Общий volume рынка не отвечает на главный вопрос: **можно ли было купить возле дна**.

Поэтому нас интересуют:
- `entry_volume_usdc`,
- `entry_trade_count`,
- длительность нахождения в зоне входа,
- и аналогичные метрики по зоне выхода.

### 4. Один токен может иметь несколько лебедей
Это не аномалия, а нормальная структура бинарного рынка. Плоская модель "один токен — один исход" теряет повторные рабочие события.

## Что должен дать бэктест

После наполнения `token_swans` и вспомогательных таблиц бэктест должен ответить на прикладные вопросы:

- какой минимальный уровень ликвидности в зоне входа делает сигнал торгуемым;
- какие диапазоны `possible_x` встречаются достаточно часто, чтобы на них строить стратегию;
- какие категории дают лучший EV;
- сколько событий происходит в день;
- сколько времени событие живёт в рабочей зоне входа;
- насколько полезны дополнительные признаки вроде времени до закрытия рынка, типа категории, `negRisk`, групповых рынков и т.д.

Итог этого этапа — стартовые константы для:
- `config.py`;
- логики скринера;
- риск-менеджмента;
- будущего Profit Module.

## Источники данных

### `markets`
- Источник: Gamma API `GET /markets`
- Назначение: метаданные рынка, даты, категории, ссылки между market и token

### `trades`
- Источник: `data-api.polymarket.com/trades`
- Это **единственный источник правды** для цены и ликвидности
- Используется фильтр `filterType=CASH&filterAmount=10`
- Одна запись = одна реальная сделка
- Базовые поля: `side`, `size`, `price`, `timestamp`

### `price_history`
- **Исключён из архитектуры**
- Может использоваться разве что для второстепенной визуализации или отладки, но не для аналитики стратегии

---

## Практические замечания по эксплуатации коллектора

- Пайплайн должен быть идемпотентным: любой этап можно перезапустить без дублей.
- Долгие процессы на сервере запускать через `tmux`, а не через голый `nohup`.
- Логи должны быть достаточно подробными, чтобы видеть прогресс по рынкам, ETA и точки отказа.
- Любые новые эвристики сначала проверяются на сырых данных, и только потом вносятся в постоянную архитектуру.

## Краткая формула проекта

Не "найти самый низкий price print".

А:
- найти **реальный локальный минимум**,
- подтвердить **ликвидность в зоне входа**,
- измерить **масштаб последующего отскока**,
- и понять, можно ли это повторять системно с положительным EV.
