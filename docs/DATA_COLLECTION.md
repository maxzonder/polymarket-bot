# Сбор исторических данных (Бэктестинг)

Для валидации стратегии покупки дешевых долей ("чёрных лебедей") перед написанием торгового бота необходимо собрать и проанализировать исторические данные.

## Цель первого этапа
Собрать сырой датасет закрытых рынков **без жёстких фильтров по цене и времени**, чтобы бэктест сам ответил на вопросы: при каком пороге входа, временном окне и категории EV максимальный. Константы ($0.03, 48h и т.д.) — это гипотезы. Задача Data Collector — проверить их, а не подтвердить.

## Алгоритм работы сборщика (Data Collector)

1. **Запрос данных из Gamma API:**
   - Эндпоинт: `GET /markets`
   - Параметры: `closed=true`, фильтры по `end_date_min` и `end_date_max` (например, последние 1-3 месяца).
   - Пагинация: использование `limit` и `offset` (с учётом троттлинга Cloudflare, см. [API_LIMITS.md](./API_LIMITS.md)).
   - **Никаких фильтров по цене или длительности рынка на этом шаге.** Берём всё.

   **Доступные фильтры для отсечения мусора (применять осторожно, пороги определяются бэктестом):**
   - `volume_num_min` — минимальный объём торгов (USDC). Отсекает рынки без реальной активности, но боты могут нагонять объём.
   - `liquidity_num_min` — минимальная ликвидность (USDC). Сложнее нагнать ботами чем volume — более надёжный фильтр.
   - `cyom=false` — исключить "Create Your Own Market" рынки (пользовательский спам).
   - `tag_id` — фильтр по категории/тегу (для целевых ниш: киберспорт, крипто, политика).

   ⚠️ Оптимальные пороги фильтров определяются через SQL-анализ реальных данных, не задаются заранее. Текущее значение в коллекторе: `volume_num_min=100` (подобрано эмпирически, может быть скорректировано).

2. **Сохранение в базу данных (SQLite):**
   - Сырые данные сохраняются в `polymarket_raw.db` как есть (бэкап и источник для пересчётов).

3. **Скачивание истории цен:**
   - Для **каждого токена каждого рынка** (и YES и NO) запрашивается полная история цен `GET /prices-history` за весь срок жизни рынка.
   - Сохраняется в `raw_price_history`. Без фильтров по времени.
   - **Почему все токены, а не только победитель:** нас интересует динамика цены, а не итог рынка. Токен может вырасти с $0.01 до $0.40 и упасть обратно до $0 — это TP-возможность на 40x, даже если рынок он проиграл.

4. **ETL: выявление TP-возможностей (Трансформация):**
   - Для каждого токена в `raw_price_history` найти **все локальные минимумы** (точки где цена была низкой).
   - Для каждого минимума проверить: поднималась ли цена до **N×** от этого уровня в любой момент после (до закрытия рынка).
   - Записать в `markets_summary`: `min_price_observed`, `max_spike_multiplier`, `had_tp_spike` (флаг при достижении хотя бы 5x).
   - **Пороги (entry_price, tp_multiplier, time_window) — параметры анализа, не константы.** Перебираются при бэктестинге для нахождения оптимума.

## Результат бэктеста — вывод констант для `config.py`

После сбора данных бэктест перебирает комбинации параметров и находит те, где EV максимальный:

- **Оптимальный порог входа** — при каком `entry_price` ($0.01? $0.03? $0.05?) EV наибольший
- **Оптимальное временное окно** — за сколько часов до закрытия рынка чаще всего возникают TP-возможности (6h? 24h? 72h?)
- **Оптимальный TP-множитель** — 5x, 10x, 20x — что чаще срабатывает и даёт лучший EV
- **Топ-категории по EV** — `avg(max_spike_multiplier) × rate(had_tp_spike) - (1 - rate(had_tp_spike))` по каждой категории
- **Оптимальный диапазон ликвидности** — нижняя граница, ниже которой наши ордера сами двигают рынок
- **Минимальное время в "зоне входа"** — сколько минут цена должна держаться <= порога, чтобы успеть купить (отсечь миллисекундные прострелы)

Эти значения становятся стартовыми константами в `config.py` и начальными весами для `category_ev_stats` в боевом Analyzer'е.
