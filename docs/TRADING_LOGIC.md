# Логика торгового модуля (Screener, Execution, Risk)

В этом документе описаны критические правила и архитектурные решения для торговых модулей бота. Они призваны защитить депозит от проскальзывания, задержек сети и блокировки капитала.

## 0. Структура комиссий и Gas (Fee & Gas Model)

### Gas на Polygon
Polymarket использует **gasless relayer** — пользователь не платит газ напрямую. Polygon gas не является операционной статьёй расходов бота.

### Taker Fees
Большинство рынков Polymarket **не имеют торговых комиссий** (включая Киберспорт и Региональную политику — наши приоритетные ниши).

Исключения с комиссией (с 6 марта 2026):
- **Крипто-рынки** — feeRate = 0.25, exponent = 2
- **NCAAB (колледж-баскетбол)** — feeRate = 0.0175, exponent = 1
- **Серия A** — feeRate = 0.0175, exponent = 1

Формула: `fee = C × p × feeRate × (p × (1 - p))^exponent`

Ключевое свойство: **комиссия минимальна на экстремальных ценах** (именно там мы покупаем) и **максимальна на 50%**:

| Цена токена | Эффективная ставка (крипто) |
|---|---|
| $0.01 (покупка) | ~0.00% |
| $0.03 (покупка) | ~0.01% |
| $0.15 (продажа TP) | ~0.41% |
| $0.20 (продажа TP) | ~0.64% |

**Вывод:** Вход почти бесплатный. Выход (TP) стоит < 1%. При ROI 5x-1000x — несущественно.

### Учёт в EV-формуле
Для крипто-рынков EV считается с поправкой на комиссию при продаже:
```
EV = Σ(payout × (1 - fee_rate_at_tp_price)) - Σ(stake_at_entry)
```
Для остальных рынков (без комиссии) — формула без поправки.

### Проверка `feesEnabled` в Screener
Каждый рынок из API содержит поле `feesEnabled: true/false`. Скринер должен учитывать это при расчёте минимального `Total Score` — рынки с комиссией требуют чуть более высокого EV-порога для входа.

## 1. Контроль ликвидности и проскальзывания (Slippage)
**Проблема:** В стакане по целевой цене (например, $0.05) может стоять заявка всего на $1 (20 долей). Если мы кинем рыночный ордер на $5, мы выкупим эту заявку, а остальные 80 долей купим по $0.10, $0.20 и т.д. Средняя цена входа взлетит, сломав математику "Черного лебедя".
**Решение:**
- Бот выставляет **только Лимитные ордера** (GTC / GTD) или использует флаг `postOnly = true`.
- Если используется рыночный ордер (FAK/FOK), предварительно запрашивается книга ордеров (`GET /book` или через WS) для оценки **Orderbook Depth** (глубины стакана). Ордер отправляется только если на целевом ценовом уровне есть достаточный объем (Size >= нашей ставке).

## 2. Скорость реакции (Latency)
**Clarification:** Стратегия "чёрного лебедя" — это НЕ высокочастотный трейдинг. Мы не ловим момент падения цены: мы ищем рынки, где цена **уже находится** на экстремально низком уровне ($0.01–$0.03) и там остаётся часами или днями. HFT-конкуренции в таких низколиквидных нишевых рынках нет.

**Решение:**
- Модуль `screener.py` работает через **REST API (Gamma)** с периодическим опросом раз в **5–10 минут**. Этого достаточно — цена на нишевом рынке не улетает за секунды.
- **REST Polling (`GET /orders`)** используется **только** для мониторинга уже открытых позиций: отслеживание исполнения TP-ордеров. Не для поиска новых сигналов (для них отдельный поллер `GET /markets`).
- Качество фильтра (скоринг, веса, EV) важнее скорости реакции.

## 3. Заморозка капитала и Take Profit
**Проблема:** Стратегия подразумевает ставку на маловероятные события. Почти все они проигрывают в конце. Кроме того, если бот потратит весь депозит ($100 по $5 на ставку) за день, капитал будет заморожен до конца сроков этих рынков (дней/недель). Бот будет простаивать.
**Решение:**
- **Механизм Take Profit (TP) — Динамический:**
  - Сразу после покупки доли по целевой цене (напр. $0.02) бот выставляет **первый лимитный ордер на продажу** (напр. по $0.10 для 5x) на часть объёма позиции.
  - На оставшуюся часть выставляется второй TP-ордер по более высокой цене (напр. по $0.20 для 10x) **или позиция удерживается до разрешения рынка**, если это режим охоты за большими лебедями.
  - **Trailing Stop — по умолчанию отключён в v1.** На тонком рынке Polymarket он создаёт ложное чувство защиты: синтетический стоп может не исполниться по ожидаемой цене или вообще не успеть отработать. Допустим только как редкая логика `cancel/replace` уже выставленных TP-ордеров и только если в стакане есть подтверждённый bid support.
  - Оптимальные уровни TP (5x, 10x, 20x, hold-to-resolution) и соотношение лотов **определяются по результатам бэктеста** на основе статистики raw trades и реальных выплат `$1` победителям.
- **Размер позиции (Position Sizing):** Микро-ставки от $0.001 до $0.05 на долю. Цель — не крупный единичный выигрыш, а высокая оборачиваемость. Пример planktonXD: крупнейший одиночный выигрыш $2,527 = всего 2% от суммарного профита.
- Внедрение лимитов на количество одновременно открытых позиций (не более 20% от депозита в рынке одновременно).
## 4. Система Скоринга и Самообучение (Profit Module)
**Проблема:** Жестко захардкоженные фильтры (например, "только Спорт" или "только рынки короче 3 дней") могут быстро устареть, а бот будет пропускать прибыльных лебедей в новых категориях или сливать депозит на старых.
**Решение:**
Вместо бинарных фильтров (if/else) модуль Анализатора использует **Систему Скоринга (Scoring System)**, основанную на весах.

### Как работает Скоринг (Analyzer):
Каждый новый рынок из API проходит две стадии:
1. **Hard Limits (Жесткие фильтры):** Рынок активен? Цена токена <= $0.03? Если нет — сразу отказ. Ликвидность проверяется отдельно: **предпочтение низколиквидным рынкам** ($500–$5,000 total volume), где ценообразование наименее эффективно и возможна асимметрия 1000x+.
2. **Soft Weights (Динамические веса):** Рынок получает баллы за свои характеристики.
   - `category_weight` — Приоритет нишевых категорий с высокой информационной асимметрией: Киберспорт (региональные лиги) = 1.5, Крипто-ценовые триггеры = 1.3, Региональная политика = 1.2, Крупные мейнстримные рынки = 0.6
   - `duration_weight` (Чем ближе к закрытию <= 48h, тем выше балл)
   - `liquidity_inefficiency_weight` — **Обратная** зависимость от объёма: рынки с $500–$5,000 volume получают высокий балл (там хуже ценообразование). Рынки с $50,000+ volume получают низкий балл.
   - `price_extremity_weight` — Чем ниже цена токена (0.001¢ > 0.5¢ > 1¢), тем выше балл: экстремально дешёвые токены дают максимальную асимметрию
   - `hype_weight` (Зависит от количества комментариев `comment_count`)
Если сумма баллов (`Total Score`) превышает заданный порог, рынок добавляется в Watchlist Скринера.

### Как работает Самообучение (Profit & Feedback Loop):

> ⚠️ **Критический принцип:** Стратегия предполагает, что **большинство сделок убыточны** (TP не срабатывает, позиция истекает в $0). Это нормально. Редкие победы с ROI 100x-1000x перекрывают все потери. Поэтому **Profit Module не должен оптимизировать по win rate** — иначе он придёт к выводу что все категории плохие и убьёт стратегию.

**Правильная метрика — Expected Value (EV) на категорию/параметр:**
```
EV = Σ(payout победных сделок) - Σ(stake проигрышных сделок)
```
- EV > 0 → категория прибыльна, вес растёт
- EV < 0 → категория убыточна, вес падает
- Win rate игнорируется как самостоятельная метрика

**Что хранится в БД по каждой категории/параметру:**
- `sum_payout` — суммарный возврат с победных сделок (USDC)
- `sum_loss` — суммарный потраченный stake на проигранные сделки (USDC)
- `ev_score` = `sum_payout` - `sum_loss`
- `trade_count` — количество сделок (для статистической значимости; игнорировать EV при < 20 сделок)

**Автоматическая корректировка весов:**
- *Пример 1:* Киберспорт — 95 потерь по $0.01 = -$0.95, 5 побед по 500x = +$25.00. **EV = +$24.05** → вес растёт, несмотря на 95% проигрышей.
- *Пример 2:* Мейнстримная политика — 10 потерь по $0.01 = -$0.10, 2 победы по 3x = +$0.06. **EV = -$0.04** → вес падает, несмотря на 17% win rate.
- *Пример 3:* `price_extremity_weight` (цена <= $0.005) — если EV по этому диапазону положительный, вес растёт независимо от того что 99% таких позиций сгорели.

Эта архитектура корректно отражает природу стратегии: мы продаём лотерейные билеты с положительным математическим ожиданием, а не ищем высокую вероятность выигрыша.

### Защита от переобучения (Starvation Prevention):
При правильном использовании EV вместо win rate риск "голодной смерти" снижается — бот не будет банить прибыльные категории за высокий процент потерь. Однако дополнительные механизмы всё равно нужны:

1. **Минимальный порог данных (Statistical Guard):**
   Profit Module не меняет веса по категории, пока не накоплено >= 20 сделок. До этого — используются базовые (дефолтные) веса из конфига. Это защищает от диких качаний на малой выборке.

2. **Возврат к среднему (Weight Decay / Forgetting):**
   Ежедневно все экстремальные веса стремятся обратно к базовому значению (1.0). Категория с EV < 0 не остаётся "забаненной" вечно — рынок меняется, и через некоторое время она получает второй шанс.

3. **Динамический порог входа (Adaptive Threshold):**
   Если за последние 48 часов Watchlist пустой, Анализатор снижает проходной балл (Threshold) на небольшие шаги, пока не найдёт 5-10 кандидатов. При избытке сигналов порог повышается обратно.

4. **Бюджет на "Разведку" (Epsilon-Greedy Exploration):**
   5-10% сделок совершаются намеренно вне скоринга — минимальным лотом ($0.001) на случайных рынках. Если такая сделка даёт положительный EV, Profit Module пересматривает веса категории в сторону роста.

## 5. Проблемы текущего Zigzag-анализатора и архитектура swan_analyzer.py

### Проблема 1 — Zigzag не видит финальную выплату $1 (главная)
Алгоритм `analyzer.py` считает `possible_x = exit_max_price / entry_min_price` — только по движению цены в стакане. Но на Polymarket при разрешении рынка выигравшие токены получают **$1.00 за каждый токен** вне зависимости от последней рыночной цены.

Пример:
- Токен куплен по $0.001 (0.1 цента)
- В стакане цена выросла до $0.028 → zigzag записывает `possible_x = 28x`
- Рынок закрылся YES → Polymarket выплачивает $1.00 → реальный `x = 1000x`

Из реальных данных (февраль 2026): из 8 415 "лебедей" 5 186 (62%) закрылись в пользу позиции. Из них 51 дали `real_x >= 100x` при учёте выплаты, хотя в базе они числятся как 5-30x.

**Решение:** добавить поле `resolution_x` = `1.0 / entry_min_price` если `is_winner = 1`, иначе `exit_max_price / entry_min_price`.

### Проблема 2 — Zigzag пропускает "удержанные" победы
Текущий алгоритм записывает событие ТОЛЬКО если после пика цена откатилась обратно (строка `price <= float(trades[extreme_idx]["price"]) / min_recovery`). Если токен вырос с $0.001 до $0.08 и удержался до закрытия рынка — событие **вообще не попадает в базу**, потому что откат не случился.

Это критический пропуск: именно такие "устойчивые отскоки" и есть самые чистые лебеди с высокой выплатой.

**Решение:** записывать событие не только по откату, но и по финальной цене закрытия (`last_price`) + признаку `is_winner`.

### Архитектура нового скрипта: `scripts/swan_analyzer.py`

Новый подход — не "ищи зигзаги", а "найди вход в зону дна и посчитай реальный выхлоп":

1. **Найти точку входа:** первый трейд, где цена упала ниже `entry_threshold` (например < $0.02)
2. **Посчитать ликвидность на дне:** объём USDC в трейдах пока цена остаётся < `entry_threshold`
3. **Посчитать real_x:**
   - Если `is_winner = 1` → `real_x = 1.0 / entry_min_price`
   - Иначе → `real_x = max_price_after_entry / entry_min_price`
4. **Посчитать time_to_resolution:** сколько времени прошло от входа до закрытия рынка
5. **Записать в таблицу `swans_v2`** (не удалять старую `token_swans` — параллельно)

Ключевые поля новой таблицы:
- `entry_min_price` — минимальная цена в зоне дна
- `entry_volume_usdc` — ликвидность на дне (можно войти)
- `floor_duration_seconds` — сколько времени цена сидела на дне
- `real_x` — реальный итоговый икс (с учётом выплаты $1)
- `is_winner` — рынок разрешился в нашу пользу
- `resolution_x` — `1/entry_price` если is_winner, иначе max_market_x
- `time_to_resolution_hours` — от первого трейда на дне до закрытия рынка

## 6. Симуляция торгов (Dry-Run Module)
**Задача:** Тестирование всей архитектуры (Market Discovery -> Screener -> Trader -> Profit Module) на реальных рынках (Live Data) без риска потерять средства (Paper Trading).
- **Интерцептор ордеров (Order Interceptor):** В режиме `DRY_RUN=True` модуль Execution (`order_manager.py`) не отправляет подписанные транзакции в CLOB API. Вместо этого он создает виртуальный ордер в локальной SQLite-базе `paper_trades.db`.
- **Эмуляция исполнения (Fill Emulator):** Отдельный фоновый процесс периодически опрашивает стакан `GET /book` для этих виртуальных ордеров.
  - *Эмуляция Покупки:* Если `bestAsk` на рынке падает до цены нашего виртуального лимитного ордера на покупку (например, $0.05), ордер помечается как `FILLED`, и с баланса списываются виртуальные USDC.
  - *Эмуляция Take Profit:* Сразу после "покупки" создается виртуальный лимитный ордер на продажу (например, по $0.20). Если `bestBid` на рынке поднимается до $0.20, ордер помечается как `FILLED`, и виртуальный профит зачисляется на баланс.
- **Интеграция с Profit Module:** Модуль оценки профита не делает различий между реальными и виртуальными сделками. Он точно так же анализирует PnL из `paper_trades.db` и обучает веса Анализатора.
Это позволяет запустить бота на неделю "вхолостую" и посмотреть, как он адаптируется и какой PnL покажет, прежде чем давать ему реальные API-ключи с деньгами.

## 7. Режим поиска больших лебедей (Big Swan Mode, 50x–1000x)

Если цель бота — не просто ловить частые 5x-отскоки, а **искать редкие 50x–1000x события**, логика должна отличаться от обычного TP-скальпинга.

### Главный принцип
Большой лебедь — это не просто рынок, который дал 5x после дна. Это рынок, где выполнено хотя бы одно из условий:
- `real_x >= 50x`
- `resolution_x >= 50x`
- рынок исторически часто даёт паники до экстремально дешёвых цен (`<= $0.005`) с последующим разрешением в `$1`

### Критика наивной логики
Наивный подход выглядит так: 
- ищем рынок, который **уже** торгуется по `$0.01–$0.03`
- покупаем
- продаём всё на `5x`

Это неплохая логика для частичной монетизации, но она **плохо подходит для поиска очень больших лебедей**, потому что:
- крупнейшие иксы часто живут не в "лучших 5x", а в **редких победителях до resolution**
- полный выход на `5x` убивает правый хвост распределения
- polling раз в 5–10 минут может пропустить короткую панику, если бот ждёт уже готовый сигнал вместо заранее поставленного лимитника

### Правильная архитектура для big swans
Нужны **два слоя логики**:

1. **Candidate Selection (кого вообще имеет смысл покрывать лимитками)**
   - рынок открыт
   - binary YES/NO
   - текущая цена не слишком далеко от реальности стратегии (например `current_price <= $0.30`)
   - есть минимальный живой объём/оборот, чтобы рынок не был полностью мёртвым
   - категория/паттерн имеет положительный исторический EV по большим лебедям

2. **Resting Limit Execution (как именно ловить панику)**
   - бот **не ждёт**, пока рынок уже упадёт в дно
   - бот заранее расставляет resting bid-ордера на экстремальных уровнях (`$0.001`, `$0.002`, `$0.005`)
   - паническая продажа приходит к нашим ценам сама
   - именно так ловятся самые жирные хвосты, которые polling может не увидеть вовремя

### Разделить две вещи, которые раньше смешивались
Для больших лебедей нельзя сводить всё к одному "score". Нужны как минимум **две независимые оценки**:

1. **Entry Fill Score** — насколько вероятно, что рынок вообще когда-нибудь ударится в наш resting bid
   - историческая частота провалов в зону `$0.001–$0.005`
   - скорость падения
   - время до закрытия
   - категория
   - частота панических свечей / raw-trade crashes

2. **Resolution Score** — насколько выгодно держать позицию после входа
   - доля победителей `is_winner=1`
   - `resolution_x`
   - частота real 20x / 50x / 100x+
   - наличие повторяющихся паттернов (например, Elon tweet markets, event-driven crypto triggers, нишевые sports matchups)

Идея простая: один рынок может часто падать в дно, но почти никогда не выигрывать. Такой рынок годится для коротких TP-отскоков, но не для big swan hold-to-resolution.

### Новая цель оптимизации
Для big swans бот должен оптимизировать **не общий PnL всех сделок вместе**, а отдельно:
- `EV_entry_to_tp`
- `EV_entry_to_resolution`
- `P(real_x >= 20x)`
- `P(real_x >= 50x)`
- `P(real_x >= 100x)`
- `tail_ev` — вклад правого хвоста в суммарный профит

Если категория даёт мало сделок, но почти весь суммарный профит приходит именно из редких `100x+`, бот должен **повышать** её приоритет, а не занижать из-за низкого win rate.

### Рекомендуемая логика выхода для больших лебедей
Если цель — ловить большие хвосты, нельзя продавать всю позицию на первом TP. Базовый шаблон такой:
- `20–40%` позиции — на `5x–10x` для возврата капитала
- `10–30%` — на `20x–50x`
- `30–60%` — держать до resolution, если рынок относится к big-swan positive cohort

То есть TP нужен не как полная фиксация, а как **финансирование дальнейшего hold**.

### Что бот должен учить из истории
Для обучения big-swan-режима в датасете по каждому событию нужны отдельные таргеты:
- `tp_5x_hit`
- `tp_10x_hit`
- `tp_20x_hit`
- `is_winner`
- `real_x`
- `resolution_x`
- `floor_duration_seconds`
- `entry_volume_usdc`
- `exit_volume_usdc`
- `hours_to_close_from_entry`
- `category`
- `trigger_pattern` (если есть повторяемый тип события)

На этой базе можно строить не один универсальный скоринг, а **несколько режимов**:
- `fast_tp_mode`
- `balanced_mode`
- `big_swan_mode`

### Практический вывод
Если задача — именно **поиск больших лебедей**, то приоритет должен быть таким:
1. raw trades вместо агрегатов
2. resting limit orders вместо ожидания готового сигнала
3. partial TP + moonbag вместо полного выхода на 5x
4. оптимизация по `resolution_x` и правому хвосту, а не по среднему win rate
5. отдельный scorer для "можно войти" и для "стоит держать до финала"

Иначе бот получится хорошим ловцом мелких отскоков — но плохим охотником за настоящими 100x–1000x событиями.
