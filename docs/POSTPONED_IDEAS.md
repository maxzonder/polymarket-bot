# Postponed Ideas (отложенные идеи)

Идеи, убранные из активной архитектуры для упрощения и удешевления разработки.
Основание: стратегия planktonXD доказала, что чистая математика без NLP-анализа даёт $98k+ прибыли.
Вернуться к этим идеям можно после успешного запуска базовой версии бота.

---

## 1. LLM-суммаризация описаний рынков

**Идея:** Прогонять поле `description` каждого рынка через дешёвый LLM для генерации краткой выжимки (10-15 ключевых слов).

**Зачем:** Снизить стоимость аналитики при бэктестинге — анализировать выжимку вместо полного текста.

**Почему отложили:** При текущей математической стратегии (цена + ликвидность + ниша + таймер) `description` вообще не используется в боевом скоринге. Экономить на анализе текста, которого нет — бессмысленно.

**Поля, связанные с этой идеей:**
- `description_summary` TEXT — краткая выжимка из `description`, генерируется LLM только если `is_black_swan = 1`
- `market_mechanic_type` TEXT — тип механики разрешения (`Binary Event`, `Threshold/KPI`, `Date Prediction`, `Electoral`)
- `resolution_ambiguity` INTEGER — оценка "размытости" правил от 1 до 5 (5 = очень запутанные)
- `sentiment_at_bottom` TEXT — сентимент комментариев в момент падения цены на дно

---

## 2. Semantic NLP Scoring (nlp_complexity_weight)

**Идея:** Добавить в Scoring System динамический вес `nlp_complexity_weight` — оценку LLM насколько запутаны правила разрешения рынка (1-5). Гипотеза: толпа чаще ошибается в рынках с неоднозначными юридическими формулировками, что генерирует лебедей.

**Почему отложили:** Требует LLM-вызова на каждый рынок в реальном времени (задержка + стоимость). Противоречит принципу "чистая математика = максимальная скорость".

---

## 3. ChromaDB — Векторный семантический поиск

**Идея:** Хранить эмбеддинги `description` исторических рынков в локальной ChromaDB. При появлении нового рынка делать семантический поиск: "найди 5 исторических рынков с похожими правилами разрешения" → смотреть, были ли там лебеди.

**Полная архитектура:**
1. Новый рынок → его `description` превращается в вектор
2. Запрос в ChromaDB: топ-5 семантически похожих исторических рынков
3. JOIN в SQLite: были ли у этих 5 рынков `is_black_swan = 1`?
4. Если да — новый рынок получает повышенный Weight в скоринге

**Почему отложили:** Требует ChromaDB, LLM для эмбеддингов, сложного ETL. При этом схожесть описаний ≠ схожесть ценовой динамики. Математически не обоснован.

**Референс:** Официальный `Polymarket/agents` репозиторий содержит готовые примеры интеграции ChromaDB + Langchain для именно такого RAG-подхода.

---

## 4. Langchain интеграция

**Идея:** Использовать Langchain коннекторы из `Polymarket/agents` для связи LLM с API при генерации `description_summary` и NLP-скоринга.

**Почему отложили:** Нет активных NLP-задач в боевом режиме после отказа от LLM-скоринга.

## 6. Telegram Notifier (utils/notifier.py)

**Идея:** Отправлять уведомления в Telegram по ключевым событиям коллектора и бота:
- Завершение скачивания дня (`DONE`)
- Критические ошибки (rate limit, сбой соединения)
- Переход на следующую дату при скачивании диапазона
- Исполнение TP-ордера (в боевом режиме)

**Реализация:** `utils/notifier.py` с `notify(message)` через Telegram Bot API. Токен и chat_id из `.env` (`TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`). Падает тихо (WARNING в лог) если не сконфигурирован.

**Почему отложили:** Сейчас прогресс мониторится вручную. Нужно создать бота через BotFather и прописать credentials в `.env` на сервере.

---

## 5. Мониторинг новостей и соцсетей

**Идея:** Добавить мониторинг Twitter/X по ключевым словам из описания активных рынков. Детектировать внезапный рост упоминаний как предвестник ценового движения, давая фору в секунды/минуты перед реакцией рынка.

**Почему отложили:** Добавляет внешние API-зависимости и задержку. Противоречит принципу работы исключительно с цифрами из WebSocket.

**Потенциал:** Высокий — информационная асимметрия могла бы дать реальный edge поверх чистой математики.
