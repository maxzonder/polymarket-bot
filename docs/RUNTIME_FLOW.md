# Runtime Flow (Архитектура непрерывной работы)

После завершения бэктеста и определения оптимальных констант (см. `DATA_COLLECTION.md`), бот переходит в режим боевой (или симуляционной) непрерывной работы. Архитектура построена на асинхронных, слабо связанных модулях, общающихся через очереди (asyncio.Queue).

## Общая архитектура (Producer-Consumer)

### 1. Модуль-Поставщик (Market Discovery)
**Задача:** Находить и поддерживать актуальный список рынков, которые потенциально могут стать лебедями (удовлетворяют первичным фильтрам, например: спорт, закрытие через < 3 дней, достаточный объем).
- Работает через **REST API (Gamma)**, опрашивая `/events` раз в несколько минут.
- **Фильтр по времени:** Добавляет в Watchlist рынки ЛЮБОЙ длительности, у которых до `endDate` осталось **<= 48 часов**. Длинные (30-90 дней) рынки в финальной фазе включаются наравне с изначально короткими.
- Прогоняет новые рынки через `Analyzer Module`:
  - Цена токена <= $0.03 (Hard Limit)
  - Ликвидность $500–$50,000 (предпочтение нижнему диапазону, где асимметрия выше)
  - **Приоритет по категориям:** geopolitics/politics (1.5×), crypto/weather/health (1.0×), sports (0.8×), tech (0.6×), entertainment (0.5×) — из `config.py: BotConfig.category_weights`. Null-категория = нейтральный 1.0×.
- Если рынок проходит фильтр, он добавляется во внутренний список отслеживаемых рынков (Watchlist).
- Управляет жизненным циклом Watchlist: удаляет закрытые рынки.

### 2. Модуль Скринер (Market Scanner)
**Задача:** Находить рынки, где цена **уже находится** на экстремально низком уровне и соответствует критериям стратегии.
- Работает через **REST API (Gamma)**, опрашивая `/markets` раз в **5–10 минут**. Скорость не критична — цена $0.01 на нишевом рынке стоит там часами.
- Для каждого рынка из Watchlist проверяет текущий `bestAsk` (цену продажи).
- **Событие (Триггер):** Если `bestAsk` находится в целевой зоне (например, <= $0.03), и глубина стакана достаточна (Size >= нашей ставке × 2), Скринер формирует объект `SwanSignal` и отправляет его в очередь для Трейдера.
- **REST Polling (`GET /orders`)** — используется для мониторинга уже открытых позиций (исполнение TP-ордеров). WebSockets мы избегаем для упрощения инфраструктуры.

### 3. Модуль Трейдер (Execution & Risk)
**Задача:** Принимать решения по сигналам, управлять капиталом и ордерами.
- Слушает очередь сигналов от Скринера.
- **При получении `SwanSignal`:**
  1. Проверяет свободный баланс и лимиты риска (Risk Manager).
  2. Если баланс есть, отправляет лимитный ордер на покупку (через CLOB API).
  3. В случае успешного исполнения (`MATCHED`), немедленно отправляет встречный лимитный ордер на продажу (Take Profit) по расчетной цене (например, $0.20).
- Добавляет сделку во внутренний трекер открытых позиций (Active Swans).
- Ожидает исполнения Take Profit ордера (через периодический REST polling `GET /orders`). При исполнении — высвобождает капитал, пишет лог и отправляет уведомление.

## Визуализация цикла

```text
[ REST API: /events ]
        | (каждые 5 минут)
        v
+-----------------------+
|  Market Discovery     | --> Фильтр (Время, Объем, Категория)
+-----------------------+
        | (добавление в Watchlist)
        v
+-----------------------+     [ REST API: /book ]
|  Scanner (Screener)   | <--- (Периодический опрос цены)
+-----------------------+
        | (триггер: цена <= $0.05)
        v
[ Очередь: SwanSignals ]
        |
        v
+-----------------------+     [ CLOB API: Orders ]
|  Trader (Execution)   | <---> (Buy Limit -> Sell Limit TP)
+-----------------------+
        | (Управление позициями)
        v
+-----------------------+
|  Risk & Position Mgr  | --> Баланс, Открытые ордера, PnL
+-----------------------+
```

### 4. Ежедневный пайплайн и рекалибровка (`pipeline/daily_pipeline.py`)
**Задача:** Обновление данных и рекалибровка порогов скоринга.

Запускается раз в сутки (04:00 UTC). Шаги:
1. **ingest** — дозагружает рынки за пропущенные дни (`data_collector_and_parsing.py`)
2. **analyzer** — обновляет `swans_v2` для новых дат (`--date-from`/`--date-to`)
3. **feature_mart_v1_1** — полный пересчёт (`--recompute`)
4. **ml_outcomes** — ML-метки по реальным сделкам бота
5. **rejected_outcomes** — метки по пропущенным возможностям
6. **recalibrate** — пересчёт category weights → `recommended_config.json`

**Рекалибровка:** `recalibrate_scorers.py` анализирует `ml_rejected_outcomes` и `ml_outcomes`, предлагает только изменения `category_weights`. Легаси threshold-рекомендации удалены вместе со старыми config knobs.

## Архитектура Анализатора (Scoring System)
Для обеспечения гибкости (добавления новых условий) и учета весов от `Profit Module`, структура Анализатора должна базироваться на системе скоринга:

1. **Базовые фильтры (Hard Limits):** Рынки, не проходящие эти фильтры (например, ликвидность < $500), отсекаются сразу.
2. **Динамические веса (Soft Weights):** Каждый оставшийся рынок оценивается по набору фичей:
   - `category_weight` (из `config.py`: geopolitics/politics=1.5, crypto/weather/health=1.0, sports=0.8, tech=0.6, entertainment=0.5, null=1.0)
   - `duration_weight` (Чем ближе к закрытию, тем выше балл)
   - `liquidity_weight` (Чем ниже ликвидность в разумных пределах, тем выше неэффективность)
3. **Total Score:** Сумма весов. Если `Total Score > Threshold` (Порога, заданного в конфиге), рынок попадает в фокус.

Такая архитектура позволяет легко добавлять новые условия (просто новый Weight в функцию скоринга), а `Profit Module` будет автоматически крутить эти веса на основе реального PnL.
