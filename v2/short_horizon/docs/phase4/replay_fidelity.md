# Phase 4: Replay Fidelity Contract

Date: 2026-04-23
Issue: #123

This document explains the replay fidelity framework introduced in Phase 4. It defines how to capture a live run, how to replay it, how to diff it with the comparator, and what is (and isn't) guaranteed by the fidelity contract.

## Capture

To audit a live execution offline, you need to capture its runtime inputs and venue responses.

You can capture any live shell run (synthetic, dry_run, or live) by passing the `--capture-dir` flag:
```bash
python3 -m v2.short_horizon.short_horizon.live_runner <db_path> --capture-dir <bundle_dir_path>
```

At shutdown, the live runner writes a complete "replay bundle" to that directory:
- `manifest.json`: Configuration, run metadata, and artifact counts.
- `events_log.jsonl`: The exact sequence of normalized events seen by the strategy.
- `market_state_snapshots.jsonl`: A convenience extraction of market states.
- `venue_responses.jsonl`: A recording of all request/response pairs made by the execution client.
- `orders_final.jsonl` / `fills_final.jsonl`: The terminal state of all orders and fills at the end of the run (used for diffing).

## Replay

A capture bundle can be replayed entirely offline. The replay runner injects a deterministic `ReplayClock` and replaces the live venue client with a `CapturedResponseExecutionClient`. 

To replay a bundle:
```bash
python3 -m v2.short_horizon.short_horizon.replay_runner <bundle_dir> <output_db_path>
```
*Note: Replay always writes its outputs to a fresh SQLite database to avoid clobbering live run data.*

The `CapturedResponseExecutionClient` matches `place_order` requests by `client_order_id` and `cancel_order` requests by `venue_order_id`. If the strategy emits an intent that wasn't in the capture trace (or vice versa), it raises a `ReplayFidelityError` and fails the run loudly.

## Diffing

After replaying, use the comparator tool to diff the replay output database against the terminal artifacts captured in the bundle:

```bash
python3 -m v2.short_horizon.short_horizon.replay.comparator <bundle_dir> <output_db_path>
```

The comparator diffs four categories of decision outputs:
1. **Order Intents:** Match on market, token, level, entry price, and notional USDC.
2. **Skip Decisions:** Match on the reason why an intent was skipped.
3. **Cancel Intents:** Match on the venue order ID and cancel reason.
4. **Terminal Outcomes:** Match on the final execution state and filled quantity.

If the sequences match, it exits `0`. If there are any divergences, it prints a structural diff and exits `1`.

## The Fidelity Contract

### What IS Guaranteed (In-Scope)
Fidelity is strictly **decision-level**. If a captured run is replayed with the same strategy code, we guarantee the exact same observable decision sequence.
- Every `OrderIntent`, `SkipDecision`, and `CancelOrder` must fire in the exact same order, for the exact same reasons, triggered by the exact same events.
- The terminal states of all orders generated by these decisions must match the live run's recorded end states.

### What IS NOT Guaranteed (Out-of-Scope)
We do not guarantee byte-for-byte log alignment or identical telemetry.
- **Log timestamps:** Wall-clock log emission times will differ between live and replay.
- **Ephemeral retries:** If the venue connection dropped and the live client retried an HTTP call silently, replay ignores the drop and just yields the final successful venue response.
- **Same-timestamp ordering noise:** Events arriving at the exact same millisecond might be processed in a different micro-order within that time bucket, as long as the macro sequence of decisions remains identical.

Phase 5 probe artifacts are expected to pass this exact comparator gate. If a micro-live probe diverges during replay, it is considered a fidelity regression.
